# 18. 4Sum

[:paperclip: LeetCode Problem Description](https://leetcode.com/problems/4sum/description/)

## Solution: Sorting and Two Pointers (Optimal with Pruning)

The problem is a direct generalization of **3Sum**. By sorting the array and using two nested loops to fix the first two numbers (`nums[i]` and `nums[j]`), we reduce the problem to **Two Sum II** on the remaining part of the array using a two-pointer scan.

To pass extreme/large test cases efficiently in an interview, we must also implement **pruning (bound checking)**.

### Thought Process

1.  **Sorting**: Sort `nums` in ascending order.
2.  **Nested Loops**: Fix `nums[i]` and `nums[j]` where `j > i`:
    *   **Duplicate Handling**: Skip duplicate values for `i` and `j` to ensure unique quadruplets.
    *   **Pruning (Smallest Sum)**: If the smallest possible sum starting at `i` (or `j`) exceeds `target`, break early (since all subsequent sums will be even larger).
    *   **Pruning (Largest Sum)**: If the largest possible sum starting at `i` (or `j`) is less than `target`, skip to the next iteration (since even the largest numbers cannot reach `target`).
3.  **Two Pointers**: Run a two-pointer scan between `l = j + 1` and `r = n - 1`:
    *   If `sum == target`: Append the quadruplet. Increment `l`, decrement `r`, and skip all duplicate values for both pointers.
    *   If `sum > target`: Decrement `r`.
    *   If `sum < target`: Increment `l`.

### Go Code

``` go
import "sort"

func fourSum(nums []int, target int) [][]int {
    sort.Ints(nums)
    n := len(nums)
    var result [][]int
    if n < 4 {
        return result
    }
    
    for i := 0; i < n-3; i++ {
        // Skip duplicates for the first element
        if i > 0 && nums[i-1] == nums[i] {
            continue
        }
        
        // Pruning 1: The smallest possible sum starting at i is too large
        if nums[i] + nums[i+1] + nums[i+2] + nums[i+3] > target {
            break
        }
        
        // Pruning 2: The largest possible sum starting at i is too small
        if nums[i] + nums[n-3] + nums[n-2] + nums[n-1] < target {
            continue
        }
        
        for j := i + 1; j < n-2; j++ {
            // Skip duplicates for the second element
            if j > i+1 && nums[j-1] == nums[j] {
                continue
            }
            
            // Pruning 3: The smallest possible sum starting at j is too large
            if nums[i] + nums[j] + nums[j+1] + nums[j+2] > target {
                break
            }
            
            // Pruning 4: The largest possible sum starting at j is too small
            if nums[i] + nums[j] + nums[n-2] + nums[n-1] < target {
                continue
            }
            
            l, r := j+1, n-1
            for l < r {
                sum := nums[i] + nums[j] + nums[l] + nums[r]
                if sum == target {
                    result = append(result, []int{nums[i], nums[j], nums[l], nums[r]})
                    l++
                    r--
                    
                    // Skip duplicates for the left pointer
                    for l < r && nums[l] == nums[l-1] {
                        l++
                    }
                    // Skip duplicates for the right pointer
                    for l < r && nums[r] == nums[r+1] {
                        r--
                    }
                } else if sum > target {
                    r--
                } else {
                    l++
                }
            }
        }
    }
    
    return result
}
```

### Code Efficiency

- **Time Complexity**: $O(n^3)$
    - Sorting takes $O(n \log n)$ time. The nested loops of depth 3 take $O(n^3)$ in the worst case. However, the pruning checks make the average execution time significantly faster than a raw $O(n^3)$ scan.
- **Space Complexity**: $O(\log n)$ or $O(1)$
    - Depending on Go's sorting implementation, we require $O(\log n)$ auxiliary space. The output slice space is determined by the number of valid quadruplets.
